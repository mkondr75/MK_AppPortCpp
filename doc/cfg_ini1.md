* Жизненный цикл конфигурации

Утверждённые решения:

* глобальная конфигурационная структура допустима;
* ConfigManager отвечает только за сериализацию/десериализацию;
* SystemProbe является исполнительным слоем проверки;
* пользовательские решения приоритетнее автоматических ограничений;
* при ошибках и повреждениях запуск переводится в мастер настройки;
* тихие recovery-сценарии запрещены;
* формат конфигурации: JSON;
* runtime-state не сериализуется в конфигурацию;
* быстрый probe используется при штатном запуске;
* активный probe используется при ошибках, первом запуске и диагностике.

# 1. Цель

Разработать подсистему начальной инициализации приложения, обеспечивающую:

* чтение и запись `app.ini`;
* валидацию конфигурации;
* обработку первого запуска;
* обследование системы;
* интерактивный мастер настройки;
* сохранение подтверждённой конфигурации;
* безопасный запуск основного приложения;
* диагностику состояния запуска;
* журналирование причин отказа и перехода в режим настройки.

Под диагностикой состояния запуска понимается:

* фиксация стадий запуска;
* фиксация ошибок bootstrap;
* фиксация решений пользователя;
* фиксация результатов probe;
* возможность определить причину перехода в мастер настройки.

Подсистема должна работать независимо от основной бизнес-логики приложения.

# 2. Основные сценарии запуска

## 2.1 Нормальный запуск

Условия:

* `app.ini` существует;
* конфигурация читается;
* все обязательные параметры валидны.

Последовательность:

1. Чтение `app.ini`.
2. Парсинг.
3. Валидация.
4. Запуск основного приложения.

---

## 2.2 Первый запуск

Условия:

* `app.ini` отсутствует.

Последовательность:

1. Создание временной конфигурации по умолчанию.
2. Обследование системы.
3. Запуск мастера первичной настройки.
4. Получение подтверждённых параметров.
5. Финальная проверка.
6. Сохранение `app.ini`.
7. Запуск основного приложения.

---

## 2.3 Повреждённая или неполная конфигурация

Условия:

* `app.ini` найден;
* часть параметров отсутствует или невалидна.

Поведение:

* автоматическое silent-recovery запрещено;
* любые ошибки или неоднозначности переводят систему в мастер настройки;
* пользователь принимает итоговое решение;
* приложение может рекомендовать варианты, но не блокирует подтверждённый пользователем выбор;
* проблемные пункты отображаются как блокирующие до явного решения пользователя;
* корректные пункты отображаются как подтверждённые.

# 3. Архитектурное разделение

## 3.1 Основные модули

### ConfigManager

Ответственность:

* загрузка JSON-конфигурации в структуру;
* сохранение JSON-конфигурации из структуры;
* сериализация;
* десериализация.

Ограничения:

* не выполняет системные проверки;
* не запускает процессы;
* не валидирует среду исполнения;
* не принимает решений recovery.

---

### ConfigValidator

Ответственность:

* проверка обязательных полей;
* проверка диапазонов;
* проверка структуры конфигурации;
* интерпретация результатов SystemProbe;
* формирование диагностического результата.

Особенности:

* validator не ограничивает пользовательский выбор;
* validator может выдавать предупреждения;
* validator может переводить запуск в мастер настройки;
* окончательное решение остаётся за пользователем.

---

### SystemProbe

Назначение:

* исполнительный слой проверки среды.

Ответственность:

* выполнение системных проверок;
* запуск probe-команд;
* получение версий компонентов;
* проверка файлов и путей;
* возврат результатов ConfigValidator.

Пример:

```text
node --version -> err / vXX.XX.X
```

Правила:

* probe фиксирует фактическое состояние;
* probe не принимает архитектурных решений;
* probe не ограничивает пользователя;
* probe возвращает факты и ошибки.

Режимы работы:

### Fast Probe

Используется при штатном запуске.

Проверки:

* существование файлов;
* существование путей;
* быстрые проверки доступности.

### Active Probe

Используется:

* при первом запуске;
* при ошибках;
* при диагностике;
* при переходе в мастер настройки.

Проверки:

* запуск процессов;
* получение версий;
* активные проверки среды.

---

### FirstRunWizard

Ответственность:

* UI мастера настройки;
* навигация между шагами;
* редактирование конфигурации.

---

### BootstrapController

Ответственность:

* управление стадиями запуска;
* принятие решения:

  * запуск приложения;
  * запуск мастера;
  * аварийное завершение.

# 5. Формат app.ini

## 5.1 Вариант A — WinAPI INI - точно нет

Использование:

* `GetPrivateProfileStringW`
* `WritePrivateProfileStringW`

Плюсы:

* минимальная реализация;
* нативность.

Минусы:

* слабая диагностика;
* ограниченная расширяемость.

---

## 5.2 Вариант B — собственный парсер JSON

Утверждённый вариант.

Требования:

* полный контроль сериализации;
* диагностика ошибок;
* Unicode;
* расширяемость структуры;
* поддержка миграции формата;
* человекочитаемый формат;
* отсутствие runtime-данных внутри конфигурации.

## 5.3 JSON — комментарии и ручное редактирование

Принято:

* конфигурация ориентирована прежде всего на приложение;
* ручное редактирование допускается;
* комментарии выносятся в help/readme;
* отсутствие inline-комментариев считается допустимым.

Отложено:

* выбор JSON-библиотеки;
* поддержка JSON5/комментариев.

# 11. Поток запуска

Черновой псевдокод:

```cpp
int WINAPI WinMain(...) {

    AppConfig cfg;

    if (!LoadConfig(L"app.ini", cfg)) {

        cfg = GetDefaultConfig();

        ProbeResult probe = ProbeSystem();

        RunFirstRunWizard(cfg, probe);

        SaveConfig(L"app.ini", cfg);
    }
    else {

        ValidationResult vr = ValidateConfig(cfg);

        if (!vr.ok) {

            ProbeResult probe = ProbeSystem();

            RunFirstRunWizard(cfg, probe, vr);

            SaveConfig(L"app.ini", cfg);
        }
    }

    return RunMainApplication(cfg);
}
```
